# 재고 조정 메커니즘 설명

## 개요

재고 조정은 두 가지 방식으로 동작합니다:
1. **재고 입고 (INBOUND)**: 실제 컨테이너 수량 증가
2. **재고 소모 (CONSUMPTION)**: 판매 이력만 생성 (실제 수량은 유지)

## 데이터 구조

### 컨테이너 실제 수량
- **테이블**: `tb_container`
- **컬럼**: 
  - `co_trade_bales`: 무역 베일 수량, `co_sales_bales`: 영업 베일 수량(없으면 무역과 동일)
  - `co_weight`: 실제 중량 (kg)
- **의미**: 물리적으로 창고에 있는 실제 수량

### 판매 이력
- **테이블**: `tb_sales_item`
- **컬럼**:
  - `si_cargo_bales`: 판매된 베일 수량
  - `si_cargo_weight`: 판매된 중량 (kg)
- **의미**: 판매/소모된 수량 기록

### 가용 수량 계산
```
가용 베일 = 컨테이너 영업 베일 (co_sales_bales 또는 co_trade_bales) - 판매된 베일 합계
가용 중량 = 컨테이너 실제 중량 (co_weight) - 판매된 중량 합계
```

## 재고 조정 동작 방식

### 1. 재고 입고 (INBOUND)

**동작**:
1. 컨테이너의 실제 수량 증가
   ```typescript
   container.bales = currentBales + inputBales
   container.weight = currentWeight + inputWeight
   ```
2. 판매 이력 생성 (이력 추적용)
   - 상태: `INVENTORY_INBOUND`
   - 재고 상태 계산에서 제외 (실제 수량 증가이므로)

**결과**:
- ✅ 컨테이너 실제 수량 증가
- ✅ 판매 이력에 기록
- ✅ 가용 수량 증가 (실제 수량 증가, 판매 수량은 변하지 않음)

**예시**:
- 현재: 베일 100개, 중량 2000kg
- 입고: 베일 10개, 중량 200kg
- 결과: 베일 110개, 중량 2200kg (실제 수량 증가)

---

### 2. 재고 소모 (CONSUMPTION)

**동작**:
1. 컨테이너의 실제 수량은 변경하지 않음
2. 판매 이력만 생성
   - 상태: `SALES_ITEM_COMPLETED`
   - 고객: 없음 (null)
   - 재고 상태 계산에 포함 (판매 수량으로 계산)

**결과**:
- ❌ 컨테이너 실제 수량 변경 없음
- ✅ 판매 이력에 기록
- ✅ 가용 수량 감소 (실제 수량은 유지, 판매 수량 증가)

**예시**:
- 현재: 베일 100개, 중량 2000kg
- 소모: 베일 10개, 중량 200kg
- 결과: 
  - 실제 수량: 베일 100개, 중량 2000kg (변경 없음)
  - 가용 수량: 베일 90개, 중량 1800kg (판매 수량 차감)

---

### 3. 판매 항목 수정 (재고 조정 항목만)

**동작**:
1. 컨테이너의 실제 수량은 변경하지 않음
2. 판매 항목의 `cargoBales`, `cargoWeight`만 수정
3. 재고 상태 재계산

**결과**:
- ❌ 컨테이너 실제 수량 변경 없음
- ✅ 판매 이력 수정
- ✅ 가용 수량 재계산 (판매 수량 변경 반영)

---

### 4. 판매 항목 삭제 (재고 조정 항목만)

**동작**:
1. 컨테이너의 실제 수량은 변경하지 않음
2. 판매 항목 상태를 `SALES_ITEM_CANCELLED`로 변경
3. 재고 상태 재계산

**결과**:
- ❌ 컨테이너 실제 수량 변경 없음
- ✅ 판매 이력 삭제 (취소 처리)
- ✅ 가용 수량 증가 (판매 수량 감소)

---

## 현재 구현의 특징

### 재고 입고 (INBOUND)
- **실제 수량 증가**: 물리적으로 창고에 추가된 수량 반영
- **이력 기록**: 언제, 얼마나 추가되었는지 기록

### 재고 소모 (CONSUMPTION)
- **실제 수량 유지**: 물리적으로 창고에 있는 수량은 그대로
- **가용 수량 감소**: 판매 수량으로 계산하여 가용 수량만 감소
- **이력 기록**: 언제, 얼마나 소모되었는지 기록

## 사용 시나리오

### 시나리오 1: 실제 창고에 추가 입고
- **사용**: 재고 입고 (INBOUND)
- **결과**: 실제 수량 증가, 가용 수량 증가

### 시나리오 2: 실제 창고 수량과 시스템 수량 불일치 (실제로는 없음)
- **사용**: 재고 소모 (CONSUMPTION)
- **결과**: 실제 수량 유지, 가용 수량 감소 (시스템 수량 조정)

### 시나리오 3: 재고 조정 항목 수정/삭제
- **사용**: 판매 항목 수정/삭제
- **결과**: 실제 수량 유지, 가용 수량 재계산

## 주의사항

1. **재고 입고 (INBOUND)**는 실제 물리적 수량 증가를 의미합니다.
2. **재고 소모 (CONSUMPTION)**는 실제 물리적 수량 감소가 아니라, 시스템상 가용 수량만 감소시킵니다.
3. 실제 창고 수량과 시스템 수량이 불일치할 때는 **재고 소모**를 사용하여 시스템 수량을 조정합니다.
